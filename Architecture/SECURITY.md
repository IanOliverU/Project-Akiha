# SECURITY.md — Project Akiha

Two separate concerns: (A) the app must not become an actual attack vector, (B) the packaged binary must not get falsely flagged as malware on Windows. Both matter starting now, even while this is personal-only — habits formed early are what hold once it's public.

## (A) Runtime security rules

- **AI output never executes directly.** The AI can only produce a `Command` proposal. Execution always passes through `PermissionGate`. This defends against prompt injection (e.g. from a summarized PDF or webpage) turning into an unwanted system action.
- **Least privilege by default.** Permissions are per-capability (file access, app launching, clipboard, screen/window reading) — never one blanket "allow automation" toggle.
- **Plugins are untrusted code, always.** No exceptions for your own test plugins — build the habit now. Real isolation (subprocess + IPC) required before any third-party plugin is ever loaded; in-process Python restriction is not a real boundary.
- **No shell string interpolation.** Build argument lists for `subprocess`, never f-string a command together — even for "trusted" internal calls.
- **No embedded secrets.** API keys/tokens live in user config or environment variables only. Never committed, never hardcoded, never bundled in the binary.
- **Validate all external input** — plugin manifests, config files, anything read from disk that isn't generated by the app itself — against a strict schema before use.
- **Don't log raw conversation content by default.** Treat chat/memory data as sensitive from day one.
- **Encrypt local data at rest if/when it matters to you** — SQLCipher is a reasonable option; optional for a personal project, worth reconsidering before public release.

## (B) Avoiding false-positive AV/SmartScreen flags on Windows

- **Code signing is the biggest lever.** Unsigned .exe from an unknown publisher is what triggers most SmartScreen warnings and AV heuristic flags. Budget a standard code signing certificate before any public release — not needed now, but design the packaging pipeline assuming it's coming.
- **Prefer Nuitka over PyInstaller.** PyInstaller's `--onefile` mode (self-extracting at runtime) closely resembles how real malware droppers behave and is a well-known false-positive trigger. Nuitka compiles to real C and generally scores better. If PyInstaller is used, prefer `--onedir` over `--onefile`.
- **No packing/obfuscation.** UPX compression and code obfuscation are both AV red flags, even though they feel like "extra security." Ship plain compiled output.
- **Expect early flags regardless — this is normal.** Reputation (SmartScreen and many AV engines) builds over time with a consistent signing identity and download volume. Don't panic-react to the first flag on a freshly signed binary.
- **Submit to Microsoft's false-positive portal** for the first few public releases if flagged.
- **Run every public release through VirusTotal before shipping** — catch an obvious vendor false-positive before users do.
- **Document what the app does and doesn't access**, plainly, in the README/UI — for an app category that's inherently "reads your desktop + runs an AI," transparency about scope is a trust factor as much as a technical one once strangers are installing it.

## Checklist before any public release

- [ ] Code signing certificate applied
- [ ] Packaged with Nuitka, no UPX/obfuscation
- [ ] VirusTotal scan clean (or false positives documented/submitted)
- [ ] All automation actions gated through `PermissionGate` with audit logging
- [ ] Plugin loading uses subprocess/IPC isolation, not in-process trust
- [ ] No secrets in repo history or binary
- [ ] README documents exactly what data is read/stored and what automation can do
